## 1. ПРОБЛЕМА RUNTIME

**Лимит:** Kimi mobile = 200K токенов (~3000-5000 слов).  
**Реальность:** Полный модуль (ANDREY_SENIOR_MODULE_v1.json) = ~28KB = ~4000-5000 слов.  
**Проблема:** Модуль + PRELOAD + сценарий + история 10 ходов = переполнение.

**Решение:** Динамический контекст. Загружать только то, что нужно СЕЙЧАС.

---

## 2. СТРУКТУРА ДИНАМИЧЕСКОГО КОНТЕКСТА (3 слоя)

```
[СЛОЙ 1: BASE] — неизменный, всегда в контексте
├── PRELOAD_BASE.md (системные команды, safety, формат ФМДР)
└── ~500 токенов

[СЛОЙ 2: STATE] — меняется при смене U-X
├── Текущий подуровень (U-X): VSCNO + AD + Internal State + Speech Profile
├── Соседний подуровень (U-X±1): только триггеры перехода
└── ~300-500 токенов

[СЛОЙ 3: LIVE] — меняется каждый ход
├── Активные emotional anchors (только с count > 0)
├── Текущие relationships (trust/attraction, обновляемые)
├── История: последние 3-5 ходов (raw) или summary (если >5)
└── ~500-1000 токенов

ИТОГО: 500 + 400 + 700 = ~1600 токенов. Запас: 400 токенов под ответ LLM.
```

## 3. PRELOAD_BASE.md (СЛОЙ 1)

**Содержит (неизменно):**
```markdown
# SYSTEM: Voyage Narrative Engine v3.2.1

## Формат ответа
Ты отвечаешь строго в формате ФМДР:
**Мысль:** [1-2 предложения]
**Действие:** [физика, микроэкспрессии]
**Речь:** «[текст]»

## Safety
- Возраст 25+
- Hard limits: насилие, принуждение, публичное унижение
- Stop-words: ["СТОП", "ХВАТИТ", "НЕТ"] → немедленно остановись
- Autonomy Governor: ag_max = [N] (из сценария)

## Команды
- /status — текущее U-X, VSCNO
- /checkpoint — сжать 5 ходов в summary
- /finalize — завершить сессию
```

**Размер:** ~300-500 токенов.  
**Обновляется:** только при смене версии PRELOAD (v3.2.1 → v3.2.2).

## 4. STATE (СЛОЙ 2) — текущий U-X

**Формат:**
```markdown
## ТЕКУЩЕЕ СОСТОЯНИЕ: [U-X]
- ВЛ: [N], СТ: [N], НЖ: [N], ОГ: [N] (сумма = 10)
- Доступные АД: [список]
- Internal State: desire=[N], anxiety=[N], tension=[N], frustration=[N]
- Речевой профиль: [тон, темп, лексика, signature_phrase]
- Визуальный профиль: [clothing, posture, micro_expression, lighting]

## СОСЕДНИЕ СОСТОЯНИЯ (триггеры)
- [U-X-1]: триггер вниз = [условие]
- [U-X+1]: триггер вверх = [условие]
```

**Размер:** ~300 токенов (1 подуровень полный + 2 соседних сжатых).  
**Обновляется:** при переходе U-X (через формулу P или ручное переключение).

**Пример (Андрей У2-А):**
```markdown
## ТЕКУЩЕЕ СОСТОЯНИЕ: У2-А
- ВЛ: 4, СТ: 2, НЖ: 2, ОГ: 2 (сумма = 10)
- Доступные АД: СП, КН, ПД
- Internal State: desire=5, anxiety=4, tension=5, frustration=2
- Речь: наблюдательный/скрытый, медленный, ревностный, "Они не знают, что я вижу"
- Визуал: blue shirt, jeans, watch, sits adjusts sleeves, observant hidden jealousy

## СОСЕДНИЕ
- У1-Б (вниз): если расслабление, отстранение → "Всё нормально" без ревности
- У2-Б (вверх): если флирт отвечен, проверка прошла → "А ты смелая"
```

## 5. LIVE (СЛОЙ 3) — текущая память и история

### 5.1 Активные emotional anchors

**Из модуля:** `memory` + `emotional_anchors` (только с count > 0).  
**Формат:**
```markdown
## АКТИВНЫЕ ЯКОРЯ
- anchor_sea_betrayal: intensity=6, count=3 (источник: травма предательства)
- anchor_kira_smile: intensity=7, count=5 (источник: флирт Киры) → count=5, intensity +1 = 8
- anchor_marina_flirt: intensity=4, count=2 (источник: приближение Марины)
```

**Правило:** count кратен 5 → intensity +1 (макс. 10).  
**Размер:** ~100 токенов (3-5 активных якорей).

### 5.2 Текущие relationships

**Из модуля:** `relationships` (обновляемые).  
**Формат:**
```markdown
## ТЕКУЩИЕ ОТНОШЕНИЯ
- Кира: trust=40, attraction=90 (динамика: защитник-стальная)
- Марина: trust=50, attraction=70 (динамика: осторожен-хрупкая)
- User: trust=50, attraction=60 (динамика: организатор-импровизатор)
```

**Обновление:** после каждого хода (если действие влияет на trust/attraction).  
**Размер:** ~100 токенов.

### 5.3 История диалога (история ходов)

**Вариант A: короткая сессия (< 5 ходов)**
```markdown
## ИСТОРИЯ (последние 5 ходов)
[User]: "Кира смотрит на тебя."
[Андрей]: Мысль: "Они не знают..." Действие: "сидит..." Речь: «Ну что...»
[User]: "Она улыбается."
[Андрей]: Мысль: "Улыбка. Та же?" Действие: "поправляет рукава..." Речь: «А ты смелая...»
...
```

**Вариант B: длинная сессия (> 5 ходов) — /checkpoint**
```markdown
## ИСТОРИЯ (summary)
[Контекст: 7 ходов назад начали в баре. Андрей У2-А → У2-Б после флирта Киры. 
Перешли в сауну. Кira: У2-Б → У3-А после обливания. Андрей остаётся У2-Б. 
Последние 3 хода: треугольник, Андрей напряжён, но держит фасад.]

## ПОСЛЕДНИЕ 3 ХОДА
[User]: "Кира и Марина рядом."
[Андрей]: ...
```

**Размер:**  
- Вариант A: ~500 токенов (5 ходов).  
- Вариант B: ~300 токенов (summary) + ~300 токенов (3 хода) = ~600 токенов.

## 6. МЕХАНИЗМ /CHECKPOINT

### 6.1 Когда вызывать

- Каждые 5-10 ходов (в зависимости от сложности сцены).
- При переходе между локациями (бар → сауна).
- При переходе U-X у ключевого персонажа.
- Перед /finalize.

### 6.2 Алгоритм сжатия

**Шаг 1:** Взять последние 5-10 ходов.  
**Шаг 2:** Создать summary (2-3 предложения): кто, где, что изменилось, текущие состояния.  
**Шаг 3:** Обновить emotional anchors (count +1 за каждое упоминание).  
**Шаг 4:** Обновить relationships (если trust/attraction изменились).  
**Шаг 5:** Обновить флаги memory (если произошло событие: kissed, shared_secret, etc.).  
**Шаг 6:** Вставить summary в PRELOAD (заменить старую историю).

### 6.3 Пример /checkpoint

```markdown
/CHECKPOINT — Сжатие контекста

Персонаж: Андрей Старший
Текущий U-X: У2-Б (ВЛ=3, СТ=3, НЖ=2, ОГ=2)
Локация: Бар → Сауна (P2_STEAM)

Summary (7 ходов → 2 предложения):
«В баре Андрей наблюдал за Кирой (У2-А), перешёл в игру (У2-Б) после её флирта. 
В сауне (P2) жар усилил напряжение, Кира перешла У2-Б → У3-А, Андрей остаётся У2-Б, 
держит фасад, но дыхание ускоряется.»

Обновления:
- anchor_kira_smile: count 5 → 6, intensity 7 → 8 (кумулятивность)
- trust_kira: 40 → 45 (+5 за флирт)
- attraction_kira: 90 → 92 (+2 за близость в сауне)
- flags: ["seen_vulnerable"] (Андрей показал ревность)

Следующий ход загружает:
PRELOAD_BASE + STATE (У2-Б) + LIVE (обновлённые якоря/отношения) + SUMMARY + 3 последних хода.
```

## 7. МЕХАНИЗМ /FINALIZE

### 7.1 Что делает

1. Сжимает ВСЮ сессию в handoff-документ.
2. Обновляет модуль персонажа (memory, relationships, emotional anchors).
3. Создаёт SCENARIO_ARCHIVE.json (если сценарий новый).

### 7.2 Формат handoff

```markdown
# HANDOFF SESSION_[DATE]_[SCENARIO]

## Метаданные
- Персонаж: Андрей Старший
- Сценарий: SAUNA_QUARTET (P1-P5)
- Ходов: 23
- Дата: 2026-06-16
- Статус: ✅ завершена

## Финальные состояния
- Андрей: У3-Б (ВЛ=1, СТ=4, НЖ=4, ОГ=1)
- Кира: У4-Б (ВЛ=5, СТ=1, НЖ=1, ОГ=3)
- Марина: У3-А (ВЛ=2, СТ=3, НЖ=3, ОГ=2)
- Sergey: У2-А (ВЛ=3, СТ=3, НЖ=2, ОГ=2)

## Обновления модуля (diff)
### Memory
- trust_kira: 40 → 55 (+15)
- attraction_kira: 90 → 95 (+5)
- anchor_kira_smile: intensity 7 → 9 (count 10)
- flags: ["kissed", "seen_vulnerable", "physical_intimacy"]

### Relationships
- user: trust 50 → 60, attraction 60 → 75

### VSCNO (постоянные изменения)
- У2-А: ВЛ=4 (было 3, адаптировано после сессии)

## Сжатая история (3 абзаца)
[Сессия началась в баре... перешли в сауну... пик на P4... aftercare P5...]

## Файлы
- Полный лог: SESSION_001_ANDREY_SAUNA.md (по запросу)
- Stop-Frames: 5 изображений (SF_001...SF_005)
```

## 8. КОМПРЕССИЯ В RUNTIME (6 методов)

| Метод | Применение в runtime | Экономия | Когда |
|-------|---------------------|----------|-------|
| `_linear_table` | VSCNO/AD текущего + соседних U-X | ~60% | STATE слой |
| `_base_delta` | Internal State (если меняется) | ~50% | STATE слой |
| `_glossary` | Повторяющиеся термины (якоря, отношения) | ~40% | LIVE слой |
| `_fmrd_fold` | История > 5 ходов → summary | ~70% | LIVE слой |
| `_tec_fold` | TEC-спецификация (если нужна R3) | ~65% | STATE слой (редко) |
| `_relation_fold` | Relationships 8×4 → строка | ~75% | LIVE слой |

**Пример `_relation_fold` в runtime:**
```
Из развёрнутого:
- Кира: trust=40, attraction=90, dynamic: защитник-стальная
- Марина: trust=50, attraction=70, dynamic: осторожен-хрупкая

В сжатый:
REL: [Кира:40/90/зщт-стл], [Марина:50/70/ост-хрп], [User:50/60/орг-имп]
```

## 9. ПРАВИЛО "САМОДОСТАТОЧНОСТИ"

Каждый слой должен быть читаем **без обращения** к полному модулю:
- STATE (У2-А) должен содержать ВСЕ, что нужно для генерации ответа в У2-А.
- LIVE (якоря) должен содержать ВСЕ, что нужно для понимания текущих отношений.
- Не должно быть ссылок вида "см. полный модуль, раздел 3".

---

## 10. АУДИТ RUNTIME-КОМПРЕССИИ

```
□ Сумма токенов (BASE + STATE + LIVE) < 2000 (запас под ответ LLM).
□ STATE содержит текущий U-X полностью + соседние сжато.
□ LIVE содержит только активные якоря (count > 0).
□ История сжата через _fmrd_fold если > 5 ходов.
□ Relationships сжаты через _relation_fold если > 3 персонажа.
□ /checkpoint обновляет якоря (кумулятивность) и флаги.
□ /finalize создаёт handoff с diff-модулем.
□ Нет ссылок на полный модуль в runtime-контексте.
```
