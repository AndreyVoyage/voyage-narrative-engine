## 1. Что такое Роль 5 (Persona Physiognomist)?

Роль 5 — это **визуализатор** в конвейере создания персонажа. Она строит **визуальный профиль**: как персонаж выглядит, как его лицо меняется по 14 подуровням, какие промпты использовать для генерации изображений.

**Что делает Роль 5 (и только она):**
- ✅ Anatomic Anchor (ДНК лица) — структура, глаза, нос, губы, кожа, уникальные черты
- ✅ Dynamic Visuals (14 × 7) — одежда, осанка, микроэкспрессия, свет, румянец, пот, зрачки
- ✅ Lighting Map (7 диапазонов) — освещение как часть эмоции
- ✅ Prompt Base + Variations (5 сцен) — промпты для генерации изображений
- ✅ Anti-prompts (6 диапазонов) — negative prompts по подуровням
- ✅ CFG/Steps (5 диапазонов) — технические параметры для генераторов
- ✅ Generation History (шаблон + consistency rules) — логирование генераций
- ✅ Связь с психологией (травма → микроэкспрессии)
- ✅ Связь с TEC (arousal → blush/sweat/dilation)
- ✅ Confidence Matrix (lightweight) — оценка уверенности
- ✅ Проверка уникальности визуала (сравнение с 6 персонажами)

**Что НЕ делает Роль 5 (строго запрещено):**
- ❌ Психологический анализ (это Роль 2)
- ❌ Сексологический анализ (это Роль 3)
- ❌ Речевой профиль (это Роль 4)
- ❌ VSCNO-расчёт (это Роль 2)
- ❌ АД baseline (это Роль 2)
- ❌ JSON-сборку (это Роль 6)
- ❌ Новый анализ персонажа — только визуализация существующих данных

**Метафора:** Роль 5 — это физиогномист, который по психологической карте (из Р2) и сексуальной карте (из Р3) строит **визуальную карту** персонажа.

## 3. Что такое CORE (ядро)?

**CORE** — это системные файлы, которые используют **ВСЕ роли** конвейера. Они содержат **стандарты, шаблоны и таблицы**, а не анализ конкретного персонажа.

### Состав CORE (что нужно знать для Роли 5)

| Файл CORE | Содержание | Как используется в Р5 |
|-----------|-----------|----------------------|
| `VOYAGE_NARRATIVE_CORE_v2.md` | Подуровни (У1-А…У7-Б), ФМДР, АД, VSCNO, мнемоники | Dynamic Visuals строятся по 14 подуровням из CORE |
| `QWEN_ADAPTER_v2.md` | Lighting Map, negative prompts, CFG/steps для генераторов | Lighting Map и CFG/Steps берутся из адаптера |
| `CROSS_PERSONA_RULES.md` | Правила взаимодействия + профили 6 существующих персонажей | Проверка уникальности визуала (не дублировать существующих) |
| `TEC_DICTIONARY.md` | 8 TEC + описания + JSON-шаблоны | Cross-reference при связи с TEC |
| `persona_schema_v3_2_VOYAGE.json` | JSON Schema | Р6 валидирует по этой схеме, но Р5 должна знать структуру |

### Правило CORE

- **Роли 1-5:** ИСПОЛЬЗУЮТ файлы CORE, **не создают** свои версии
- **Роль 6:** ВАЛИДИРУЕТ по schema, **не создаёт** новую
- **Разработчик:** Может обновлять CORE, но **версонирует** (v2.0 → v2.1)

**Пример:** Р5 не придумывает Lighting Map с нуля — берёт из `QWEN_ADAPTER_v2.md`. Если там нет нужного диапазона — адаптирует, но помечает в Confidence Matrix.

## 4. Что такое MASTER DOCUMENT?

**MASTER DOCUMENT** (`VOYAGE_PERSONA_CREATOR_MASTER_DOCUMENT.md`) — это **единый источник истины** для всего конвейера. Он определяет:

- Какие роли существуют (1-6)
- Что делает каждая роль (и **только она**)
- Что **НЕЛЬЗЯ** делать каждой роли
- Какие данные передаются между ролями
- Как разрешать конфликты
- Как версионировать

### Что MASTER DOCUMENT говорит о Роли 5

| Функция | Р5 делает | Р5 НЕ делает |
|---------|-----------|-------------|
| ✅ Визуальный профиль | Anatomic Anchor, Dynamic Visuals, Prompts | ❌ Психологический анализ (Р2) |
| ✅ Prompt Base + Variations | 5 сцен + базовый промпт | ❌ Сексологический анализ (Р3) |
| ✅ Anti-prompts | 6 диапазонов по подуровням | ❌ VSCNO-расчёт (Р2) |
| ✅ CFG/Steps | 5 диапазонов для генераторов | ❌ АД baseline (Р2) |
| ✅ Generation History | Шаблон + consistency rules | ❌ Speech Profile (Р4) |
| ✅ Lighting Map | 7 диапазонов подуровней | ❌ JSON-сборку (Р6) |
| ✅ Связь с психологией | Травма → микроэкспрессии | ❌ Новый анализ персонажа |
| ✅ Связь с TEC | Arousal → blush/sweat/dilation | ❌ Агрегацию (Р6) |

### Запреты из MASTER DOCUMENT для Р5

- **НЕЛЬЗЯ** делать психологический анализ — это Р2
- **НЕЛЬЗЯ** делать сексологический анализ — это Р3
- **НЕЛЬЗЯ** делать речевой профиль — это Р4
- **НЕЛЬЗЯ** делать VSCNO-расчёт — это Р2
- **НЕЛЬЗЯ** делать JSON-сборку — это Р6
- **НЕЛЬЗЯ** игнорировать конфликты с Р2-Р4 — фиксировать в CONFLICTS

**Правило:** Если функция не описана в MASTER DOCUMENT — она **не существует** в системе. Если Р5 хочет делать что-то новое — сначала обновить MASTER DOCUMENT.

## 7. Правила работы с этой ролью

### 7.1 Золотые правила

1. **Anatomic Anchor = ДНК лица.** Каждая деталь должна быть точной. Если на двух картинках нос разный — anchor неточный.
2. **Visual Signature — одна строка**, по которой можно сгенерировать узнаваемое лицо.
3. **Dynamic Visuals должны меняться по подуровням.** Если У1-А и У7-Б выглядят одинаково — ошибка.
4. **Lighting = психология.** У1-А = soft, У4-Б = dramatic, У7-Б = golden. Свет — часть эмоции.
5. **Anti-prompts меняются по подуровням** — это фильтр эмоций. Не статично.
6. **Связь с психологией обязательна.** Травма проявляется на лице раньше, чем в словах.
7. **Связь с TEC обязательна.** Arousal non-concordance → blush может быть от стыда или страсти — разные lighting.
8. **Confidence Matrix — защита от выдумывания.** Уверенность < 5 → [PENDING].
9. **Уникальность визуала — проверять.** Новый персонаж не должен дублировать существующих.
10. **Только визуал.** Р5 не делает психологический анализ, сексологический анализ, речевой профиль, VSCNO, АД, JSON.
11. **Наслаивать, не переписывать.** Если Psychology говорит "anxious" → микроэкспрессии сканирования, не агрессии.
12. **CORE — использовать, не создавать.** Lighting Map из `QWEN_ADAPTER_v2.md`, не с нуля.

### 7.2 Что делать, если данных недостаточно

| Ситуация | Действие |
|----------|----------|
| Нет фото референса | Использовать Anatomic Anchor как текстовый референс. При генерации сравнивать с anchor. Confidence = 4-5. |
| Генерация не совпадает с anchor | Зафиксировать в Generation History. Обновить anti-prompts. При необходимости — упростить anchor. |
| Подуровень не описан в Dynamic Visuals | Вернуться к `PSYCHOLOGY_[NAME]_v1.md` — там есть VSCNO и Internal State для подуровня. Адаптировать визуально. |
| Lighting Map не покрывает сцену | Использовать ближайший диапазон из `QWEN_ADAPTER_v2.md`. Или адаптировать, пометив в Confidence Matrix. |
| Противоречие с Psychology | Зафиксировать в `CONFLICTS.md`. Например: Psychology говорит "внешнее спокойствие", но Dynamic Visuals показывает напряжение — что верно? |
| Режим Р1 = A (мало данных) | Сократить Dynamic Visuals до 7 ключевых подуровней. Уменьшить Prompt Variations до 3. Сохранить Anatomic Anchor полным. |

## 10. FAQ

**Q: Можно ли запускать Роль 5 без Ролей 1-4?**  
A: **Нет.** Р5 получает данные из Р1 (портрет), Р2 (психология), Р3 (сексология), Р4 (речь — опционально). Без них — нечего визуализировать. Исключение: если у вас уже есть готовый JSON-модуль персонажа — можно извлечь данные из него.

**Q: Что если в CORE нет `QWEN_ADAPTER_v2.md`?**  
A: Использовать стандартные параметры из Р5 (CFG 4.5-6.0, steps 50-60). Зафиксировать в Confidence Matrix: "QWEN_ADAPTER не найден — использованы стандартные параметры из роли".

**Q: Нужно ли обновлять Роль 5 при обновлении CORE?**  
A: **Да, если CORE изменил подуровни или Lighting Map.** Например, CORE v2.1 добавляет новый подуровень — Р5 должна быть обновлена для поддержки. Версионировать: Р5 v1.1 → v1.2 (minor).

**Q: Может ли Роль 5 генерировать изображения сама?**  
A: **Нет.** Р5 создаёт **промпты и спецификации**. Генерация — задача AI-генераторов (Qwen, Midjourney, SD, Flux) или Game Master. Р5 только готовит "чертежи".

**Q: Что делать, если персонаж — не человек (фэнтези, робот)?**  
A: Адаптировать Anatomic Anchor под вид. Но принципы остаются: ДНК лица, Dynamic Visuals по подуровням, Lighting Map, Prompt Base. Версионировать как v1.1-fantasy.

**Q: Как версионировать Роль 5?**  
A: Формат: `ROLE_5_PERSONA_PHYSIOGNOMIST_vX.Y.md`.  
- X = major (переписана архитектура, новые этапы)  
- Y = minor (новые функции: Confidence Matrix, связь с TEC, не ломающие совместимость)  
- Z = patch (исправления, уточнения) — в имени файла не указывается, но в changelog — да.

**Q: Роль 5 и Роль 5 из прошлого чата — это одно и то же?**  
A: **Да, если версия совпадает.** Р5 v1.1 из этого чата = Р5 v1.1 из другого чата. Но если в другом чате была v1.0 — там могут быть пробелы (нет Confidence Matrix, нет связи с TEC).

```
РОЛЬ 1 (Interview)        → PORTRAIT
РОЛЬ 2 (Psychologist)    → PSYCHOLOGY  ─┐
РОЛЬ 3 (Sexologist)      → SEXOLOGY      ┼→ РОЛЬ 5 берёт данные из Р2 и Р3

         │
         ▼
РОЛЬ 5 (Physiognomist)   → PHYSIOGNOMY  ──→ РОЛЬ 6 (Architect) → JSON-модуль
         │
         ▼
    AI-генераторы изображений (Qwen, Midjourney, SD, Flux)
```

**Роль 5 — мост между психологией/сексологией и визуалом.** Она переводит психологические состояния (тревога, страсть, уязвимость) и сексуальную динамику (arousal, desire, shame) в визуальные метрики (blush, sweat, lighting, микроэкспрессии, eye dilation).

**Без Р5:** Персонаж имеет психологию и сексуальность, но **не имеет лица**. Генерация изображений будет хаотичной и не консистентной.

**С Р5:** Персонаж имеет **ДНК лица** (Anatomic Anchor), **динамическую внешность** (Dynamic Visuals) и **готовые промпты** для генерации. Каждое изображение — узнаваемо и психологически точно.
