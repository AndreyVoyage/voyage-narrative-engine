## 2. Два вида ролей — не смешивать

- **Dev-роли** (`.voyage/roles.yaml`): безопасность разработки — кто может писать код/доки, какие пути
  разрешены/запрещены, кто аудирует. Не «думают» как сценарист.
- **Нарративные роли** (`roles/`, R1–R8 и др.): психолог персонажа, лингвист, архитектор характера,
  автор диалогов, аудитор уникальности речи, редактор канона, visual/video prompt director.

Эта граница уже зафиксирована в самом `roles.yaml`; документ её лишь наследует.

## 3. Эволюция ролей (единственная реально новая механика)

Роль со временем специализируется, но **не меняет свой prompt сама**:

```
Роль выполняет задачи
  → система собирает результаты и ошибки
  → ретроспектор анализирует качество
  → формируется ПРЕДЛОЖЕНИЕ новой версии
  → человек проверяет
  → появляется новая версия (старая остаётся для сравнения и отката)
     ROLE_DIALOGUE_WRITER_v1 → v1.1 → v2
```

**Инвариант:** роль формирует *предложение* новой версии, а не переписывает себя. Совпадает с моделью
proposals+approval из N7 §7. Версии неизменяемы, откат возможен.

## 4. Per-role knowledge profile (динамический роутинг)

Не «загрузить всю базу знаний каждой роли», а:

```
роль → определяет информационную потребность
     → Knowledge Router выбирает источники
     → Gateway отдаёт только нужные модули (allowlist + provenance)
     → роль работает с ограниченным контекстом
```

Примеры профилей:
- **Психолог:** `personas/<char>/psychology/`, `knowledge_base/R2/` (KB_R2_CORE, KB_R2_AD_RULES, KB_R2_VSCNO_RULES — роль R2 Persona Psychologist), история отношений.
- **Автор диалогов:** speech matrix, характерные фразы, уровень сцены, последние реплики, эмоц. состояние.
- **Аудитор канона:** INDEX персонажа, provenance/content_hash, активные решения, проверяемый текст.
- **Visual director:** NCC visual canon, approved reference pack, scene request, camera/clothing guardrails.

Поиск: **литеральный/токенный по модулям сначала**; semantic/embeddings — отложено (согласуется с N7,
где embeddings вынесены за MVP).

## 5. Как это связано с остальным (integration)

```
PAC                 → вызывает роли: автор, психолог, аудитор речи
Character Evolution → вызывает роли: психолог, аналитик эволюции, аудитор узнаваемости
Persona Gateway     → отдаёт канонические модули (общая точка доступа)
Knowledge Router    → решает, какие модули нужны конкретной роли (тонкий слой над Gateway)
N8                  → в будущем использует одобренный корпус, но НЕ управляет ролями
Ren'Py runtime      → отдельный потребитель, не источник авто-эволюции ролей
```

## 6. Guardrails

1. Роль **не** переписывает свой prompt автоматически (только предложение → человек).
2. Канон (`personas/`, `core/`, `knowledge_base/`, `scenarios/`, `v2_*`) — только чтение.
3. **Нет утечки знаний между персонажами:** профиль роли ограничен нужным персонажем (Gateway per-character).
4. Роль **не** читает PAC session memory и Sandbox state без отдельного решения (§7 D-RKR-11/12).
5. Provenance использованного контекста логируется (что именно роль прочитала).

## 7. Owner decisions — PENDING (D-RKR)

Пока не приняты — все `OWNER_DECISION_PENDING`. Условно D-RKR, не N-номер (номер присваивать отдельно).

```
D-RKR-1   Какие роли входят в первый реестр?
D-RKR-2   Где канонический role prompt (roles/ vs иное)?
D-RKR-3   Кто выбирает роль под задачу: человек / router / гибрид?
D-RKR-4   Может ли одна задача запускать несколько ролей последовательно?
D-RKR-5   Knowledge profile каждой роли (разрешённые каталоги/модули)?
D-RKR-6   Semantic search или сначала точный модульный? (реком.: модульный сначала)
D-RKR-7   Как ограничивается размер контекста?
D-RKR-8   Как измеряется качество роли?
D-RKR-9   Как утверждается новая версия роли? (реком.: proposal → human)
D-RKR-10  Как выполняется rollback?
D-RKR-11  Может ли роль читать PAC memory? (реком.: нет по умолчанию)
D-RKR-12  Может ли роль читать Sandbox state? (реком.: нет по умолчанию)
D-RKR-13  Как предотвращается утечка знаний между персонажами?
D-RKR-14  Где живёт будущий Knowledge Router? (реком.: тонкий слой над Gateway)
D-RKR-15  Как сохраняется provenance использованного контекста?
```
