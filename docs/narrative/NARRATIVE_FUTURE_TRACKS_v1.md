## ТРЕК A — Character Aside (non-canonical private chat)

### Что это
Во время игры игрок ставит сценарий на паузу и общается наедине с выбранным персонажем в отдельной
LLM-сессии («параллельный мир»). Персонаж знает текущий контекст истории, ведёт живой разговор, затем
игрок возвращается ровно в ту же точку сценария. Канон не меняется.

- Технический термин: **Character Aside — non-canonical private chat**
- Игровой/UI термин: **«Поговорить наедине»** / «Параллельный разговор»
- Это самый безопасный первый слайс LLM-слоя (write-back выключен by design).

### Модель памяти (ядро идеи)
1. **Canon memory** (`player_state`: flags, `character_states`, `relationships`, проигранная история) —
   подаётся В чат как контекст (только чтение).
2. **Ephemeral aside memory** — живёт только внутри сессии, НЕ сливается обратно в `player_state`.

Формула:
```text
Персонаж в private chat помнит игру.
Персонаж в игре не помнит private chat.
```

### Persistent aside memory (уточнение, 2026-07-02)
По решению: aside-память может быть **персистентной между сессиями** (не только эфемерной). Тогда две
параллельные памяти на персонажа:
```text
1. Канон-таймлайн (player_state) — ведёт игру; двигается только внутрисценными выборами.
2. Aside-таймлайн (приватная память параллельного мира) — копит прошлые разговоры; НЕ сливается в канон.
```
Персонаж в aside видит: канон до текущего момента (двигается вперёд) + прошлые приватные разговоры.
В игре — по-прежнему только канон (инвариант «в игре не помнит чат» держится: игра читает канон, не aside).

Технические следствия:
- **Хранение:** прошлые транскрипты (`private_chats/<char>/session_NNN.json`) подгружаются в контекст aside.
- **Рост контекста → суммаризация/компакция** (сводка прошлых бесед + последние реплики), иначе не влезет.
- **Save:** aside-память входит в сейв, но в **отдельный namespace**, не в канонический player_state
  (расширение STORY_RUNTIME_CONTRACT §8: сейв хранит прогресс + некононическую aside-память). Привязка
  к прохождению/сейву, по персонажу.
- **Правило конфликта:** для фактов истории авторитетен канон; aside-память — приватный эмоциональный
  слой. При расхождении персона приоритезирует канон для «что произошло».
- **Границы неизменны:** aside → aside_memory → будущий aside; НЕ пишет во flags/relationships/scenario/player_state.
- **Опционально:** сброс приватной памяти персонажа (некононично, безопасно).

Фазы: v1 Character Aside = эфемерная (без персистентности); **v2 = персистентная aside-память + суммаризация**.
Обе внутри N6.

Терминология (принято 2026-07-02): вместо «эфемерная память» — **Persistent non-canonical aside memory,
isolated from canon**. Игровое имя: «Память разговоров наедине».

**Односторонняя связь памяти:**
```text
canon_state НЕ читает aside_memory.
aside_memory читает canon_state.
```

**Форма данных (ориентир):**
```json
{
  "canon_state": { "scene_id": "...", "branch": "...", "beat_id": "...", "flags": [], "relationships": {}, "character_states": {} },
  "aside_memory": {
    "kira": {
      "summary": "краткая сводка прошлых приватных бесед",
      "sessions": [ { "scene_id": "...", "beat_id": "...", "summary": "...", "non_canonical": true } ]
    }
  }
}
```

**Защита от «памяти из будущего» (важный guardrail).** Aside-память привязана к `save_slot` + канон-позиции:
```text
save_slot_id, scene_id, branch_id, beat_id, story_progress_index
Правило: персонаж помнит только те aside-разговоры, что произошли НЕ ПОЗЖЕ текущей канон-точки.
Откат save назад → память из будущего в чат НЕ подаётся.
```

**Три режима памяти** (настройка/dev-config):
```text
OFF        — каждый parallel chat забывается после выхода.
SESSION    — помнит внутри текущей игровой сессии, не между запусками.
PERSISTENT — помнит прошлые parallel-разговоры в рамках save slot.   ← рекомендуемый
Рекоменд.: PERSISTENT per save slot / per character (Кира помнит своё, Марина своё, Сергей своё).
```

**Хранение (layout, отдельно от канона):**
```text
private_chats/
  save_slot_001/
    kira/   { memory_summary.json, sessions/SC_017_B03_session_001.json, ... }
    marina/ { memory_summary.json, sessions/... }
```

**Сборка контекста для LLM** (чтобы не раздувать):
```text
1) current canon snapshot (до текущего beat)
2) character persona
3) aside memory summary
4) последние 10–20 сообщений текущего чата
```

**In-fiction рамка** (мягко объясняет игроку изоляцию): «здесь, наедине» / «в этом тихом месте между
событиями» / «когда время остановилось» / «там, куда не дотягивается основной мир». Не обязательно
говорить прямо «это неканоническая LLM-сессия».

**Итоговая формула Character Aside:**
```text
- читает канон только до текущего beat;
- хранит отдельную private memory между aside-сессиями (persistent, non-canonical);
- не пишет в canon_state; не влияет на ветки; не даёт знать будущее;
- при откате save не подаёт memories из будущего;
- может быть очищен игроком вручную.
```

### Guardrails (архитектурные запреты, не prompt-правила)
1. **Нет write-access к канону.** Character Aside не пишет в `player_state`, flags, relationships,
   scenario JSON, branch effects. Это ограничение на уровне архитектуры, а не «мы попросили LLM».
2. **Персонаж не знает будущего.** Контекст = «прошлый световой конус»: персона + `player_state` +
   уже проигранные beats. Будущие beats, другие ветки и невыставленные флаги в контекст НЕ подаются,
   хотя runtime их знает. Нужна отдельная функция проекции состояния (context projection = past-only).
3. **Никаких обещаний, ломающих канон.** Персонаж говорит в настоящем времени о чувствах, не даёт
   обещаний о будущем, которое сценарий потом опровергнет.
   - Нельзя: «Я больше никогда не встречусь с Сергеем.»
   - Можно: «Сейчас мне хочется держаться от него дальше, но я сама запуталась.»
4. **Транскрипты хранятся отдельно** и не входят в канонический save:
   ```text
   private_chats/SC_017/kira/session_001.json
   ```
5. **Save-as-candidate — только в карантин, только явным авторским действием.** Хорошие фрагменты не
   идут в канон и не идут напрямую в Director. Они уходят в карантинную область:
   ```text
   aside_candidates/   (или director_inbox/)
   ```
   → позже отдельная нормализация в V2 → только после review может стать каноном. Никогда автоматически.
6. **Мост памяти (imprint) — по умолчанию OFF, и даже во включённом виде структурно изолирован.**
   Если когда-нибудь добавим imprint-эффекты (`player_understands_kira_better`, `optional_memory_token`),
   они живут в отдельном неструктурном namespace (напр. `meta.aside_*`), и каноническим веткам
   ЗАПРЕЩЕНО на них ветвиться. Они могут влиять только на флейвор/опциональный контент, не на
   разрешение веток.
7. **Safety наследуется.** Свободная взрослая LLM-генерация → действуют платформенные фильтры +
   tone/safety-правила (`RN-SAFETY-STYLE-1`) + `content_rating`/границы из настроек игрока. Канон это
   не гейтит (там ничего не сохраняется), но генерацию — да.
8. **Выбор персонажа поэтапно.** Первая версия — только доступные в текущем состоянии персонажи; позже
   расширить до introduced-but-absent (напр. Сергей при `present:false`).

### Технический поток
```text
Story Runtime → snapshot (scene_id, branch, current_beat, past flags/state, relationships, past context)
Private Chat Layer → временный LLM-диалог (персона + snapshot)
Main Game Runtime → после выхода продолжает игру с того же beat_id, канон не тронут
```

### Зависимости и место в roadmap
- Требует: N1 (ссылки/состояние персонажей), N2 (`player_state`/effects/branch playback — **уже есть**),
  и **live pause/resume by beat_id — это будущий контракт, ещё НЕ реализован** (важная поправка:
  N2 не даёт beat-level resume «из коробки», его нужно формализовать), и желательно N4 (UI-режимы).
- Место: **N6 / LLM Character Layer**, отдельным модулем «рядом с runtime», не внутри обычного runtime.
- НЕ N4. Сейчас в код и документы репозитория не вносить.

## ТРЕК B — Audio / Voice Layer

### Базовый принцип
Голос — это **render-слой поверх text-first источника**, не источник правды. Аудио — производный
артефакт из текста (`speech` в beat'ах / текстового ответа LLM). Источник правды остаётся текстом в JSON.

```text
Voice output must never become canonical state.
(TTS не пишет flags / relationships / player_state / scenario JSON / branch effects.)
```

### Два разных случая
**1. Canon Voice Assets — озвучка сценарных сцен.**
```text
dialogue beat (speech) → TTS на этапе авторинга → audio asset (ключ = beat_id)
→ ручная проверка → RenPy проигрывает voice
```
- Генерация офлайн, разово; игроку НЕ нужен мощный компьютер во время игры.
- Детерминированно, кэшируется. Ложится на renderer-слой (расширение N3-пути «beat → asset»).
- Тайминг: среднесрочно, после стабильного renderer/runtime. **Не N4.**

**2. Aside Dynamic Voice — озвучка приватного чата.**
```text
LLM answer as <персонаж> → TTS в голосе персонажа → playback в private chat
→ транскрипт/аудио НЕ пишутся в канон
```
- Нужен TTS в реальном времени: локальный или облачный.
- Место: внутри **N6** (вместе с Character Aside).

### Local vs Cloud
- **Cloud** (ElevenLabs / Azure / Google / OpenAI TTS и т.п.): проще, качественнее, низкая задержка,
  без локального железа. Минусы: постоянная плата, нужен интернет, и **контент-политики** — многие
  облачные TTS ограничивают откровенный контент и могут отказать. Удобно для прототипа.
- **Local** (модель на GPU): приватность, офлайн, без платы за использование, автономность. Минусы:
  нужно железо (GPU для хорошего качества/скорости) и настройка. **Для aside с взрослым контентом
  долгосрочно предпочтительнее** (приватность + отсутствие контент-политик).

### Guardrails
1. **VoiceProvider — абстракция**, чтобы не привязывать игру к одному сервису:
   ```text
   VoiceProvider → { cloud | local | mock | cached }
   ```
   Смена провайдера без переписывания игры.
2. **Voice output never canonical** (см. выше).
3. **Voice cloning реальных голосов — только при наличии прав/согласия.** Безопаснее синтетические
   или лицензированные голоса; единый голос на персонажа — фиксированный synthetic или клон по
   разрешённому референсу.
4. **Конкретику проверять ближе к реализации.** Сервисы, цены, лимиты, latency, content policy быстро
   меняются — на этапе выбора нужно перепроверять; сейчас фиксируем интерфейс, а не сервис.

### Место в roadmap
- Отдельный поздний **Audio / Voice Layer**: canon voice assets — среднесрочно (после renderer/runtime);
  aside voice — внутри N6. **Не N4.**

## ТРЕК C — Story Setup (pre-game route & behavior personalization)

> **РЕШЕНИЕ (2026-07-02): принят вариант «игрок выбирает ветвь сам по ходу сценария».**
> Направление истории (ревность / треугольник / отношения со всеми) задаётся **внутрисценными
> выборами** (`choice_points` → `branches` + накопление флагов), а НЕ опросом перед стартом.
> Новой архитектуры не требуется — это существующий механизм схемы/рантайма.
> - Pre-game Story Setup (опрос → маршруты/диспозиция) — ОТЛОЖЕНО как опциональная идея, не строим.
> - Presentation-настройки (режимы чтения, `dialogue_verbosity`) — остаются как тип A (подача, не канон).
> - Динамическое поведение персонажа (генерация, не выбор из авторских веток) — N6/LLM, если вообще.
> Ниже — исходная проработка идеи, сохранена для истории/возможного возврата.

### Что это
Опрос перед стартом (и/или во время игры), ответы на который формируют **канонические маршруты
истории и диспозицию персонажей**. Примеры (это НЕ сценарий, а иллюстрация направлений):
- Кира ищет внимания других мужчин → маршрут с ревностью игрока;
- игрок склоняет Киру к любовному треугольнику;
- игрок выстраивает отношения со всеми.
Плюс формат диалогов: подробный / средний / короткий, с мыслями / без / минимум.

### Ключевое различие двух видов персонализации
```text
A. Presentation personalization (уже в PLAYER_EXPERIENCE_SPEC §7) — меняет ПОДАЧУ, не канон:
   reading_mode, thought_visibility_pref, drama_intensity(подача), pacing, verbosity(отображение).

B. Story/Route personalization (ЭТОТ трек, пока НЕ задокументирован) — КАНОНИЧНА, структурна:
   ответы опроса → story-shaping флаги + character disposition → гейтят АВТОРСКИЕ маршруты/ветки.
```
Это разные слои. Сейчас доки описывают только A и прямо говорят «settings = подача, не источник
правды», поэтому B нужно вводить как отдельное понятие **Story Setup**, а не как ещё одну настройку.

### Как это работает (на существующем механизме)
```text
Pre-game answers
  → seed story-shaping flags в player_state.flags  (напр. route_jealousy, kira_seeks_attention)
  → seed character disposition в character_states   (напр. kira.disposition = seeking_attention)
Runtime (уже есть механизм):
  → доступность сцены = prerequisites ⊆ completed ∧ flags_required ⊆ flags   (STORY_RUNTIME_CONTRACT §7)
  → доступность ветки = branch condition над flags/disposition                (§5, опц. поле-расширение)
```
То есть механизм гейтинга уже описан; не хватает: (1) понятия Story Setup; (2) явного различия
story-shaping флагов от display-настроек; (3) поля branch-condition в схеме; (4) поля disposition.

### Формат диалога (verbosity)
- «С мыслями / без / минимум» — **уже покрыто** режимами чтения (`classic`=без мыслей,
  `psychological`=с мыслями) + `thought_visibility_pref`.
- «Подробный / средний / короткий» — **НЕ покрыто**. Добавить настройку:
  `dialogue_verbosity: detailed | medium | short`.
- Реализация verbosity под фиксированным текстом (дизайн-вопрос, решить позже):
  либо авторские варианты beat'ов, либо render-time обрезка, либо LLM-перефразирование (N6).

### Важное ограничение (честно)
В scripted-VN опрос **выбирает среди авторских маршрутов/диспозиций** — их надо заранее написать
(ветки под «ревность», «треугольник», «отношения со всеми»). Он **не генерирует** новое поведение
сам. Динамическая генерация поведения персонажа — позже, слой N6 (LLM).

### Guardrails
1. Story-shaping флаги (тип B) и display-настройки (тип A) — **разные namespace**, не путать.
   Канонические ветки зависят только от story-shaping флагов, не от display-настроек.
2. Опрос сеет флаги/диспозицию детерминированно; дальше всё идёт по обычному runtime-контракту.
3. Маршруты должны быть авторски написаны и провалидированы (schema/flag-lint), прежде чем опрос
   сможет их открывать.
4. Safety/границы контента (`content_boundaries`, RN-SAFETY-STYLE-1) применяются к выбранным маршрутам.

### Что нужно будет добавить в доки (после N4B, отдельными шагами)
- `PLAYER_EXPERIENCE_SPEC.md`: раздел **Story Setup** (тип B) отдельно от §7 settings; настройка
  `dialogue_verbosity`.
- `SCENARIO_SCHEMA_V2_SPEC.md`: поле условия ветки (branch condition) и поле `disposition` у персонажа.
- `STORY_RUNTIME_CONTRACT.md`: вычисление branch-condition над флагами/диспозицией.
- `NARRATIVE_ROADMAP.md`: Story Setup как продуктовая фаза (после базового runtime/UX; авторские
  маршруты — предпосылка); динамическое поведение — N6.

### Место в roadmap
Story Setup (выбор среди авторских маршрутов + verbosity) — продуктовая фаза уровня N4-позже/N5-рядом,
**после** того как есть авторские ветки под маршруты. Динамическая генерация поведения — N6. **Не сейчас.**

## Финальная сводка решений (что фиксируем)

```text
1. Character Aside и Voice Layer — будущие слои ПОВЕРХ text-first JSON, не источники правды.
2. Character Aside: read-only snapshot канона, эфемерная память, НЕТ write-back (архитектурно).
3. Персонаж помнит игру в чате; в игре не помнит чат; не знает будущего; не обещает ломать канон.
4. Транскрипты — отдельно; save-as-candidate — только в карантин, только вручную, только через review.
5. Imprint-мост: default OFF; даже ON — неструктурный namespace, ветки не зависят.
6. Safety-слой наследуется в aside (фильтры + RN-SAFETY-STYLE-1 + границы игрока).
7. Voice = render-слой; аудио производно от текста; VoiceProvider-абстракция.
8. Canon voice = офлайн-ассеты по beat_id; aside voice = realtime TTS (local предпочтителен долгосрочно).
9. Voice output никогда не канон; клонирование реальных голосов — только с правами.
10. Место: N6 (Character Aside + aside voice) + поздний Audio/Voice track (canon voice). НЕ N4.
11. live pause/resume by beat_id — будущий контракт, ещё НЕ реализован (не считать готовым).
12. Сейчас (во время N4B) код и git-репозиторий не трогаем; перенос в docs/narrative после merge N4B.
```
