# ADR: граница потоков между Ren'Py main thread и фоновыми LLM/IO-воркерами

> Формат: Architecture Decision Record. В репозитории на момент написания нет
> установленной нумерованной ADR-конвенции (`docs/decisions/` создана этим документом
> впервые) — номер не присваивается.

## Context

Character Aside (`novel/game/aside.rpy`) выполняет блокирующие HTTP-запросы к LLM
(local Ollama, потенциально cloud) из фонового потока (`renpy.invoke_in_thread`), чтобы
не замораживать главный UI-поток Ren'Py на время ожидания ответа (24–114 секунд для
локальной модели в измерениях этого проекта).

В процессе разработки дважды обнаруживался один и тот же класс дефекта:

1. Фоновая worker-функция транзитивно (не в своём собственном теле, а через синхронный
   call chain в другую функцию) вызывала `import store` — обращение к Ren'Py store из
   не-главного потока.
2. После устранения (1) — worker всё ещё транзитивно резолвил `renpy.config.savedir`
   внутри JSONL trace-writer'а, вызываемого из воркера.

Оба случая проходили мимо поверхностной проверки "функция-воркер не упоминает `store`/
`renpy.config` в своём собственном коде" — потому что нарушение было в вызываемых ею
функциях, не в ней самой.

## Decision

**Фоновые потоки в Ren'Py-интеграциях не должны читать или мутировать:**

- `store` (и любой доступ через `import store`);
- `renpy.store`;
- `renpy.config` (включая `renpy.config.savedir` и любые производные пути);
- screen variables (`renpy.get_screen_variable`/`renpy.set_screen_variable`);
- любые live Ren'Py-контейнеры (revertable list/dict/set, save-tracked объекты);
- UI state любого рода.

**Обязательная граница:**

```text
MAIN THREAD                    →  detached plain data  →  WORKER THREAD
(снимок делается один раз,        (dict/list/tuple/str/     (провайдер, файловый I/O
 непосредственно перед              int/float/bool/None       по переданному пути —
 диспетчеризацией воркера)          — ничего другого)         никогда своим резолвом)
                                                                      │
                                                          renpy.invoke_in_main_thread
                                                                      ▼
                                                          MAIN-THREAD CALLBACK
                                                          (единственное место, где
                                                           результат воркера пишется
                                                           обратно в store/UI)
```

Snapshot строится **один раз**, на главном потоке, функцией, которая явно проверяет
`threading.current_thread() is threading.main_thread()` и падает с `RuntimeError` при
нарушении. Snapshot проходит независимую рекурсивную валидацию типов перед тем, как
пересечь границу потока.

## Allowed data types

Только: `str`, `int`, `float`, `bool`, `None`, и рекурсивно построенные из них
built-in `dict`/`list`/`tuple` (со строковыми ключами словарей). Любое другое значение
(Ren'Py объект, live-контейнер, displayable, функция/callback, файловый handle) должно
быть отклонено с `TypeError` до диспетчеризации воркера, не после.

## Trace-path rule

Если воркеру требуется куда-либо писать файл (диагностика, лог, кэш) — **путь
резолвится на главном потоке** (единственная функция, которой разрешено читать
`renpy.config.savedir` для этой цели) и передаётся воркеру как plain `str` внутри
snapshot. Файловый writer, вызываемый из воркера, обязан:

- принимать путь как аргумент, никогда не резолвить его сам;
- не иметь fallback вида "если путь не передан — вычислю через `renpy.config`";
- безопасно no-op'ать (не писать, не падать) при отсутствующем/невалидном пути;
- быть обёрнутым в `try/except Exception: pass` — сбой диагностики не должен влиять на
  provider result или диспетчеризацию main-thread callback.

## Consequences

Позитивные:

- Устраняет целый класс race condition / undefined behavior, связанный с обращением к
  Ren'Py API вне главного потока (Ren'Py API в общем случае не документирован как
  thread-safe для произвольного фонового доступа).
- Делает нарушение границы **статически проверяемым**: достаточно транзитивного grep/
  call-graph аудита на отсутствие `store`/`renpy.config`/`renpy.store`/screen-variable
  паттернов в функциях, реально достижимых из воркера.
- Snapshot-валидация на границе даёт безопасный, видимый отказ (`[LLM ERROR] Cannot
  prepare worker payload: ...`) вместо тихого падения или неопределённого поведения.

Компромиссы:

- Snapshot должен явно перечислять всё, что нужно воркеру — при добавлении нового поля
  контекста (например, ещё один canon-параметр) разработчик обязан явно прокинуть его
  через snapshot, а не просто прочитать из `store` "по пути". Это дополнительный шаг,
  но он же — источник безопасности.
- Рекурсивный detach-копир не имеет explicit защиты от циклов (`visited`-set); циклическая
  структура приведёт к `RecursionError`, который перехватывается тем же error-path'ом,
  что и любая другая ошибка валидации (безопасный, но не самый диагностичный отказ).

## Alternatives rejected

- **Читать `store` из воркера "только на чтение"** — отклонено: Ren'Py store — не
  документированно thread-safe даже для чтения; наблюдаемые баги в этом проекте
  возникали именно на "безобидных" read-only путях (`_vne_readonly_canon_snapshot`,
  `renpy.config.savedir`).
- **Блокировки/mutex вокруг store-доступа из воркера** — отклонено: усложняет код без
  устранения корневой проблемы (Ren'Py UI-объекты и config всё равно не рассчитаны на
  конкурентный доступ), и требовало бы аудита каждой транзитивной зависимости на предмет
  их собственной потокобезопасности.
- **Полностью синхронный (блокирующий UI) вызов провайдера** — отклонено: при
  latency 24–114 секунд для локальной модели это неприемлемо для UX и уже было
  источником отдельного класса проблем (см. постмортем, watchdog-эксперимент).

## Test / verification requirements

Любая реализация, следующая этому ADR, должна быть проверяема:

1. **Статически** — транзитивный call-graph аудит (не только тело функции-воркера) на
   отсутствие `store`/`renpy.store`/`renpy.config`/screen-variable паттернов во всех
   функциях, реально достижимых из воркера, включая вызовы во внешние (не `.rpy`)
   модули.
2. **Runtime** — JSONL или эквивалентная инструментация, фиксирующая имя потока
   (`threading.current_thread().name`) на ключевых точках (`worker_start`,
   `provider_result`/`provider_error`, `main_thread_callback`, финальная запись
   результата), подтверждающая, что воркер-точки действительно на не-главном потоке, а
   callback/финальная запись — на `MainThread`.
3. **Независимый аудит** — второй, скептический read-only проход, не принимающий на
   веру отчёт реализующего агента; в этом проекте именно такой проход дважды находил
   транзитивные нарушения, пропущенные первичной реализацией.

## Applicability to future work

Это правило применимо к любой будущей Ren'Py-интеграции с фоновой обработкой,
включая (не ограничиваясь):

- NPC-чат вне Character Aside;
- генерация диалогов/reply suggestions в фоне;
- суммаризация памяти персонажей (аналог `aside_memory_store`, выполняемая асинхронно);
- аналитика/телеметрия, требующая сетевого вызова без блокировки UI;
- удалённые или локальные вызовы моделей вне LLM-чата (image generation, TTS, и т.п.);
- любая фоновая персистентность (autosave-подобные фичи, если они станут асинхронными).
