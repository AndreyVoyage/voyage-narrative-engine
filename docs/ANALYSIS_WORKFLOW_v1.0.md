## 2. Что сломано или отсутствует (критические пробелы)

### 🔴 Проблема 1: STATE не обновляется
**Суть:** Narrative Editor удаляет `У2-А`, `ВЛ2 СТ3` — но эти данные нужны для **следующей сессии**.

**Последствия:**
- Следующий раз вы начинаете с `U1-A`, хотя Кира уже была на `U3-Б`
- `trust_levels` не растут — персонажи «забывают» доверие
- `flags` (phone_exchanged, kissed) теряются

**Решение:** Отдельная роль **State Manager**, которая анализирует лог ДО Narrative Editor и выдаёт `STATE_UPDATE.json` + `MEMORY_UPDATE.json`.

---

### 🔴 Проблема 2: Пользовательские команды исчезают
**Суть:** В логе есть строки:
```
Пользователь: ВЛ2 СТ3
Пользователь: АД КН
Пользователь: кладу руку на талию
```
Narrative Editor v1.0 предлагает их просто удалить или превратить в действия. Но `ВЛ2 СТ3` — это **изменение атмосферы сцены**, а не действие персонажа.

**Последствия:**
- В рассказе теряется эмоциональная динамика (весёлость/сосредоточенность/негатив)
- Читатель не понимает, почему персонажи вдруг изменили поведение

**Решение:** Добавить в Narrative Editor правило: *«Мнемоники ВЛ/СТ/НЖ/ОГ превращайте в описание атмосферы, а не удаляйте»*.

---

### 🟡 Проблема 3: Нет извлечения визуальных промптов
**Суть:** В логе полно `*Кира покраснела, отводя взгляд*`, `*Марина улыбнулась, поправляя волосы*`. Это идеальные моменты для генерации картинок.

**Последствия:**
- Красивые моменты остаются только в тексте
- Для иллюстраций нужно вручную перечитывать рассказ

**Решение:** Роль **Visual Extractor**, которая из лога вытаскивает key moments и генерирует Qwen-промпты.

---

### 🟡 Проблема 4: Нет структуры хранения результатов
**Суть:** Где лежат `STORY_2026-06-08.md`? В `stories/`? В `docs/`?

**Решение:** Чёткая структура папок для артефактов сессии.

## 3. Доработанный Pipeline (3 роли вместо 1)

```
┌─────────────────────────────────────────────────────────────┐
│                    POST-SESSION PIPELINE v1.1                │
├─────────────────────────────────────────────────────────────┤
│  ШАГ 0: Сырой лог (copy-paste из чата)                      │
│         → sessions/raw/session_2026-06-08_23-49.log        │
├─────────────────────────────────────────────────────────────┤
│  ШАГ 1: STATE MANAGER (техническая роль)                    │
│  Вход: лог + текущий STATE.json + модули персонажей        │
│  Задача: найти все переходы уровней, обновить метрики      │
│  Выход: STATE_UPDATE.json + MEMORY_UPDATE.json              │
│         → sessions/state/STATE_UPDATE_2026-06-08.json      │
├─────────────────────────────────────────────────────────────┤
│  ШАГ 2: NARRATIVE EDITOR v1.1 (творческая роль)             │
│  Вход: лог + (опционально) модули для голоса               │
│  Задача: превратить лог в литературный рассказ            │
│  Выход: STORY_2026-06-08.md                                 │
│         → sessions/stories/STORY_2026-06-08.md             │
├─────────────────────────────────────────────────────────────┤
│  ШАГ 3: VISUAL EXTRACTOR (визуальная роль)                  │
│  Вход: лог + модули (visual_data)                          │
│  Задача: найти key moments, сгенерировать Qwen-промпты     │
│  Выход: VISUAL_PROMPTS_2026-06-08.md                        │
│         → sessions/visuals/VISUAL_PROMPTS_2026-06-08.md    │
├─────────────────────────────────────────────────────────────┤
│  ШАГ 4: Ручное применение (пока нет Orchestrator)           │
│  - Скопировать STATE_UPDATE.json в state/                  │
│  - Обновить personas/*/memory из MEMORY_UPDATE.json        │
│  - Сделать git commit                                       │
└─────────────────────────────────────────────────────────────┘
```

---

## 4. Почему именно 3 роли, а не 1

| Роль | Тип мышления | LLM-модель | Почему отдельно |
|------|--------------|------------|-----------------|
| **State Manager** | Аналитическое, точное | Любая (даже слабая) | Нужна математическая точность: `desire += 1`, `U1-A → U2-Б`. Творческий LLM может "приукрасить" цифры. |
| **Narrative Editor** | Творческое, стилистическое | Kimi / Claude / GPT-4 | Нужен длинный контекст и литературный стиль. |
| **Visual Extractor** | Визуальное, структурированное | Kimi / Qwen | Нужно понимание анатомических якорей и lighting_map. |

**Если совместить в одну роль:**
- LLM может забыть обновить `desire` из-за увлечённости стилистикой
- Или наоборот — сделать сухой технический отчёт вместо рассказа
- Контекст переполнится (лог + модули + правила редактуры + правила визуала)

## 5. Что остаётся ручным (пока)

| Действие | Автоматизация | Почему |
|----------|---------------|--------|
| Копирование лога из чата | Ручное | API чата недоступен |
| Применение STATE_UPDATE.json | Ручное | Нужен Python-скрипт (следующий этап) |
| Git commit | Ручное (или `git_auto.sh`) | Требует проверки человеком |
| Выбор, какие картинки генерировать | Ручное | Visual Extractor даёт 5-10 вариантов, пользователь выбирает |

---

## 6. Итог

Текущий Narrative Editor v1.0 — **хорошая база**, но он решает только задачу "красивый текст". Для полноценного цикла сессии нужно добавить **State Manager** (память) и **Visual Extractor** (картинки).
