# Context Retirement Protocol

> Общий (не специфичный для Character Aside) workflow Voyage-проекта для сворачивания
> длинной, многошаговой инженерной задачи в документацию + компактную память, чтобы
> прекратить нести детальную отладочную историю в каждой следующей сессии агента.

## Проблема

Долгая инженерная задача (расследование → фикс → независимая QA → commit → push)
накапливает десятки промежуточных гипотез, тестовых маркеров, PID, diff'ов и QA-отчётов
в рабочей памяти диалога и/или в долговременной памяти агента. После завершения задачи
эта детализация перестаёт быть нужной для дальнейшей работы, но продолжает занимать
контекст и путать последующие решения, если её не свернуть намеренно.

## Три уровня хранения

```text
HOT   — текущая активная задача; текущие блокеры; следующее действие.
        Живёт: в диалоге/памяти агента, пока задача не закрыта.

WARM  — компактная project capsule; устоявшиеся решения; ссылки на коммиты;
        ссылки на канонические документы.
        Живёт: в долговременной памяти агента, обновляется редко, только по
        завершении крупных этапов.

COLD  — полные постмортемы; сырые отчёты; JSONL/логи; скриншоты; устаревшие
        эксперименты.
        Живёт: в `docs/` (канонические выводы) и `LOCAL_STORAGE/handoffs/`
        (сырая, локальная доказательная база). НЕ в памяти агента.
```

## Жизненный цикл

```text
TASK COMPLETE
  → каноническая документация (docs/<область>/, с ADR при архитектурном решении)
  → индекс сырых доказательств (не копия, а таблица со статусами и ссылками)
  → независимый read-only аудит документации (факты сверены с финальным кодом)
  → компактная context capsule (1–2 страницы, вне репозитория)
  → selective memory retirement (удаление детальной истории из долговременной памяти)
  → новый чат (память не сокращает уже открытый диалог)
```

### Шаг 1 — каноническая документация

Создаётся в `docs/` (не в `LOCAL_STORAGE`) отдельным пакетом:

- README как точка входа с явной иерархией источников правды;
- архитектурный документ, описывающий **только финальное** состояние (не историю
  экспериментов);
- постмортем, отделяющий доказанные причины от отвергнутых гипотез;
- операционный runbook для будущей отладки аналогичных проблем;
- чеклист приёмки;
- ADR, если решение носит архитектурный, переиспользуемый характер.

### Шаг 2 — индекс сырых доказательств

Не переносить содержимое всех промежуточных отчётов в `docs/`. Вместо этого — таблица
со статусами (`CURRENT EVIDENCE` / `FINAL QA` / `HISTORICAL` / `SUPERSEDED`),
показывающая, какой отчёт что доказывает и актуален ли он ещё. Явно указать, что сырые
локальные отчёты не существуют вне конкретной машины/клона.

### Шаг 3 — независимый аудит документации

Отдельный, скептический read-only проход, сверяющий утверждения новых документов с
фактическим финальным кодом (не с текстом других документов и не с памятью автора
документации). Цель — не допустить, чтобы временный эксперимент был случайно
задокументирован как архитектура, или чтобы устаревшая гипотеза осталась
непомеченной.

### Шаг 4 — компактная context capsule

1–2 страницы, вне репозитория (`LOCAL_STORAGE/handoffs/` или аналог). Содержит только:
путь репозитория, финальную ветку/коммиты, сжатую архитектурную суть, ключевые policy-
решения (что отложено и почему), пути к каноническим документам, следующую безопасную
задачу. Не содержит: тестовые маркеры, PID (кроме явно принятых как caveat), полную
историю промптов, каждый промежуточный эксперимент, сырые лог-строки.

### Шаг 5 — selective memory retirement

Из долговременной памяти агента удаляется детальная отладочная история (см. чеклист
ниже). Остаются только устойчивые факты уровня WARM: путь репозитория, канонические
пути документации, финальные коммиты/ветка, ключевые policy-решения, ссылка на context
capsule.

### Шаг 6 — новый чат

**Удаление памяти не сокращает уже открытый диалог** — весь объём предыдущей переписки
остаётся в контексте текущей сессии до её завершения. Чтобы реально освободить
контекстное окно, необходимо начать новый чат. В новом чате достаточно приложить или
указать context capsule и путь к каноническому README пакета документации — этого
достаточно, чтобы агент восстановил рабочий контекст без повторного чтения истории
расследования.

## Memory-retirement checklist

Перед тем как просить агента "забыть" детальную историю задачи, убедиться, что:

- [ ] Канонический пакет документации создан и независимо проверен (шаги 1–3 выше).
- [ ] Context capsule создана и содержит все ссылки, необходимые для продолжения
  работы без доступа к старому диалогу.
- [ ] Финальные коммиты/ветка/push-статус зафиксированы в capsule и в
  документации согласованно (одни и те же хэши).
- [ ] Любые принятые caveat'ы (например, provenance-оговорки, отложенные файлы)
  явно перечислены как "сохранить", а не как "забыть".
- [ ] Пользователь явно сформулировал, что именно удерживать (WARM-уровень) и что
  можно удалить (детали HOT-этапа) — агент не должен сам решать эту границу без
  подтверждения.

## Что удерживать (WARM), что удалять (retire)

**Удерживать:**

- путь репозитория и рабочей директории;
- канонические пути документации;
- финальные commit hash'и и имя ветки;
- устойчивые архитектурные правила (например, содержимое принятого ADR — в сжатом виде);
- явные policy-решения (что отложено, что исключено, почему);
- ссылку на remote branch, если опубликовано.

**Удалять:**

- временные PID процессов (кроме единственного явно принятого provenance-caveat, если
  он значим для истории решения);
- тестовые маркеры (`ASIDE_*_TEST_*`-подобные строки);
- эксперименты с mirror/overlay/промежуточными layout-вариантами;
- устаревшие гипотезы, отвергнутые в ходе расследования;
- детальные построчные JSONL-события;
- повторяющиеся тексты QA-промптов, если их суть уже отражена в постмортеме;
- историю каждого промежуточного патча, если финальный diff уже задокументирован.
